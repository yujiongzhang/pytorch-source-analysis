# PyTorch Inductor 源码解析（五.二）

##  Scheduler 阶段 和 FX Passes 阶段，算子融合的区别

### 10.1 两阶段融合对比

| 维度 | FX Passes 融合 | Scheduler 融合 |
|------|--------------|---------------|
| **时机** | Lowering 之前 | Lowering 之后 |
| **对象** | FX 算子 | IR/Scheduler Node |
| **信息** | 图结构、算子类型 | 循环、内存布局、设备 |
| **决策依据** | 模式匹配 | 收益评估、backend 能力 |
| **典型融合** | Conv-BN、DDP 通信 | 点值+Reduction、Template Epilogue |
| **输出** | 修改后的 FX 图 | FusedSchedulerNode |

### 10.2 FX Passes 中已有的融合

FX passes 确实做了一些融合工作，但类型有限：

| 融合类型 | 文件 | 说明 |
|---------|------|------|
| **Group Batch Fusion** | `group_batch_fusion.py` | 批量操作融合（如多个 linear） |
| **DDP 通信融合** | `ddp_fusion.py` | 分布式通信操作融合 |
| **Attention Fusion** | `fuse_attention.py` | 注意力机制子图融合 |
| **Conv-BN Fusion** | `pre_grad.py` | 卷积 + 批归一化融合 |
| **Split-Cat Fusion** | `split_cat.py` | split + cat 抵消/融合 |
| **Binary Folding** | `binary_folding.py` | 二元操作折叠 |

**示例**（Conv-BN 融合）：
```python
# FX Passes 融合前
conv = conv2d(x, weight, bias)
bn = batch_norm(conv, running_mean, running_var)

# FX Passes 融合后
bn = batch_norm(x, fused_weight, fused_bias)  # 参数合并
```

### 10.3 为什么需要 Scheduler 阶段的融合？

#### 原因 1：FX Passes 无法访问底层循环信息

**FX 图层级**（抽象）：
```python
%y = torch.add(%x, 1)    # 只知道是加法
%z = torch.mul(%y, 2)    # 不知道迭代空间、内存布局
```

**Scheduler 层级**（具体）：
```python
SchedulerNode y: 
  sizes: [1024, 1024]
  group: (cuda, [1024, 1024])
  read_writes: MemoryDep("x", [i, j]) → MemoryDep("y", [i, j])

SchedulerNode z:
  sizes: [1024, 1024]  
  group: (cuda, [1024, 1024])
  read_writes: MemoryDep("y", [i, j]) → MemoryDep("z", [i, j])
```

**关键点**：
- Scheduler 知道**循环结构**（`[i, j]`）
- Scheduler 知道**内存访问模式**（`MemoryDep`）
- Scheduler 知道**设备信息**和**后端特性**

这些信息对融合决策至关重要：
```python
# can_fuse 需要检查
if device != device2:
    return False  # 设备不同不能融合

if shared_data_score < threshold:
    return False  # 共享数据太少，融合无收益
```

#### 原因 2：Kernel 级别的融合需要代码生成信息

**示例**：点值操作 + Reduction 融合

```python
# 原始 FX 图
%1 = torch.add(x, y)
%2 = torch.relu(%1)
%3 = torch.sum(%2, dim=1)
```

**FX Passes 无法回答的问题**：
- `add` 和 `relu` 能否在同一个 CUDA kernel 中执行？
- `sum` 的 reduction 维度是否匹配？
- 融合后寄存器压力是否会超出硬件限制？
- 融合后的内存访问模式是否高效？

**Scheduler 可以回答**：
```python
# Scheduler 知道每个节点的具体属性
node1._sizes = [[1024], [1024]]
node1._body = LoopBody(...)  # 循环结构
node1.group = (cuda, [1024])  # 迭代空间

# 可以检查
if node1.group == node2.group:
    # 相同迭代空间，可以融合
    return True
```

#### 原因 3：Template 融合需要后端支持

**示例**：GEMM Template + Epilogue 融合

```python
# FX 图
%gemm = torch.mm(x, w)
%relu = torch.relu(%gemm)
```

**Scheduler 阶段的特殊处理**：
```python
# scheduler.py:5462-5518 Prologue/Epilogue 融合检查
if node2.is_template():
    if not config.prologue_fusion:
        return False
    template = node2.get_template_node_or_throw()
    allowed_prologue_inps = template.get_allowed_prologue_inps()
    
    # 检查 node1 是否在 Template 允许的 Prologue 操作列表中
    if node1.get_buffer_names() & unsupported_prologue_args:
        return False
```

**为什么 FX Passes 做不到**：
- FX 图中 `mm` 只是一个普通算子
- Scheduler 阶段才会将 `mm` lower 为 **Triton Template**
- Template 的 epilogue 融合能力由后端（Triton/CUTLASS）决定

#### 原因 4：收益评估需要运行时信息

**Scheduler 的融合收益评估**（scheduler.py:3984-4050）：
```python
def speedup_by_fusion(self, node1, node2):
    if config.benchmark_fusion:
        # 实际 benchmark 两个节点
        ms1 = benchmark(node1)
        ms2 = benchmark(node2)
        ms_fused = benchmark(fused_node)
        
        if ms_fused < ms1 + ms2:
            return True  # 融合有收益
```

**FX Passes 无法 benchmark**：
- FX 图还没有 lower 成可执行代码
- 无法获取真实的 runtime

#### 原因 5：循环合并和重排序

**示例**：混合顺序归约融合（Mix Order Reduction）

```python
# 两个不同循环顺序的 reduction
node1: sum(x, dim=0)  # 按列归约
node2: sum(x, dim=1)  # 按行归约

# Scheduler 可以融合它们（如果满足条件）
if MixOrderReduction.can_fuse(node1, node2):
    # 需要检查循环维度、缓存行为等
    return True
```

这种融合需要：
- 知道循环的 `numel` 和 `reduction_dim`
- 知道硬件的 L2 cache 大小
- 检查是否会创建 persistent reduction

这些都是 **IR/Scheduler 层级**的信息。

### 10.4 架构设计原则

```
用户代码 (Python)
    ↓
Dynamo 捕获 (FX Graph)
    ↓
FX Passes [图级别优化]
    ├── Conv-BN 融合（参数合并）
    ├── DDP 通信融合（改变图结构）
    └── Attention 融合（子图替换）
    ↓
Lowering (IR 生成)
    ↓
Scheduler [Kernel 级别优化]
    ├── 点值操作融合（减少 kernel 启动）
    ├── Reduction 融合（减少内存访问）
    ├── Template Epilogue 融合（利用 backend 特性）
    └── Loop 重排序（优化缓存）
    ↓
Codegen (Triton/C++)
```

**核心思想**：
- **FX Passes**：改变计算图结构，做**语义等价**的图变换
- **Scheduler**：不改变计算语义，做**执行效率**优化

两者互补，而不是互相替代。
